# 银行支付通道智能路由优化方案

## 问题概述

银行支付清算系统面临一个典型的**多通道在线路由决策**问题。当前存在三条实时支付通道——大额系统（HVPS）、超网系统（IBPS）和城商行清算系统（CITY），各通道在**单笔限额**、**费率结构**和**对手行覆盖范围**上存在显著差异。当支付请求到达时，系统需要在毫秒级时间内完成通道选择，决策质量直接影响三个核心经营指标：**通道费率成本**、**头寸安全垫厚度**和**客户支付体验**。

本方案构建了一套**"预测+优化+决策"三层闭环架构**：旁路预测系统以分钟级频率滚动预测各通道未来头寸消耗趋势；实时路由引擎基于多目标加权评分模型逐笔决策最优通道；日终头寸规划模块根据预测结果优化次日各通道的初始资金配置。整个系统以运筹学中的**多目标优化**和**随机规划**为理论根基，结合梯度提升树（GBDT）的时间序列预测能力，实现了从"经验驱动"到"数据智能驱动"的通道管理升级。

## 1. 三通道业务约束与费率结构分析

### 1.1 通道业务规则梳理

三条支付通道的业务规则存在明显差异，这些规则构成了路由决策的**硬约束条件**，任何优化模型都必须优先满足。

| 维度 | 大额系统 (HVPS) | 超网系统 (IBPS) | 城商行清算 (CITY) |
|------|----------------|----------------|------------------|
| **单笔限额** | 无上限 | **必须 < 100万元** | 无上限 |
| **对手行限制** | 无限制 | 无限制 | **仅限城商行白名单** |
| **营业时间** | 工作日 8:30-17:00 | 7×24小时 | 工作日 8:30-17:00 |
| **到账时效** | 实时 | 实时 | 实时 |
| **费率结构** | 阶梯套餐+超出计费 | 阶梯套餐+超出计费 | 标准/优惠双轨阶梯 |
| **套餐档位** | 5档 (0.9万~1598万/月) | 5档 (0.9万~668万/月) | 无固定套餐 |

大额系统的核心定位是处理**100万元以上**的大额支付，虽然技术上可以处理任意金额，但对于100万元以下的支付，其费率显著高于超网通道，因此小额支付原则上应避免走大额通道。超网系统的硬性约束在于**单笔金额必须低于100万元**，这是由中国人民银行支付清算系统的技术规范所决定的。城商行通道的特殊性在于其**白名单机制**——只有对手行属于城商行名录（通过行号规则或白名单维护表识别）的支付才能通过该通道处理。

### 1.2 费率结构建模

三张费率表揭示了复杂的**阶梯定价+套餐包月**的混合计费模式。以大额系统为例，其费用由两部分构成：一是固定的**月费套餐**（包含一定笔数的信息类业务），二是按支付金额分段的**单笔处理费**。五档套餐的月费从0.90万元到1598万元不等，对应的单笔费率从3.89元/笔降至1.79元/笔（针对20万元以下档位）。

![三通道单笔费率对比](fig_fee_comparison.png)

上图清晰展示了三个通道在不同金额区间的费率差异。在**0-5万元**的小额区间，城商行清算（优惠实时贷记）和超网系统具有显著的成本优势，单笔费用可低至**0.03-0.09元**；而大额系统在此区间的费率为**3.19-3.89元/笔**，相差约两个数量级。在**5-20万元**区间，差距依然明显：大额系统3.19元/笔 vs 城商行1.17元/笔 vs 超网（优惠后）2.87元/笔。当金额超过20万元后，超网系统因触及上限而失效，大额系统与城商行通道的费率差距收窄至2-3倍。

![五档套餐费率热图](fig_fee_heatmap.png)

从热图中可以观察到两个关键规律。第一，**套餐档位越高，单笔费率越低**，这体现了规模经济效应——银行支付量越大，越应选择高档位套餐以降低边际成本。第二，**金额分段阈值对费率有跳跃式影响**，例如超网系统在5万元和20万元处出现费率跃升，大额系统在20万元和100万元处出现跃升。这些阈值点恰好是路由策略切换的"临界点"，优化模型需要精确捕捉这些非线性特征。

对于单笔支付 $a$（单位：万元）和所选套餐 $k$，通道 $c$ 的单笔费用函数可形式化为分段函数：

$$
f_c(a, k) = \sum_{j=1}^{J_c} \mathbb{1}_{[L_{c,j}, U_{c,j})}(a) \cdot r_{c,j,k}
$$

其中 $\mathbb{1}_{[L,U)}(a)$ 为区间指示函数，当 $a \in [L, U)$ 时取值为1，否则为0；$r_{c,j,k}$ 表示通道 $c$ 在第 $j$ 个金额分段、第 $k$ 档套餐下的单笔费率。此分段函数的非凸性为后续的优化求解带来了挑战，但在实际工程中可通过查表法高效计算。

## 2. 实时路由优化模型

### 2.1 问题建模

支付路由决策是一个**在线序列决策问题**。在每一决策时刻 $t$，系统收到一笔支付请求 $p_t = (a_t, b_t, s_t)$，其中 $a_t$ 为支付金额，$b_t$ 为对手行标识，$s_t$ 为业务优先级。系统需要基于当前三通道的实时状态 $S_t = (B_t^{HVPS}, B_t^{IBPS}, B_t^{CITY}, Q_t^{HVPS}, Q_t^{IBPS}, Q_t^{CITY})$，选择一个通道 $c_t \in \{HVPS, IBPS, CITY\}$ 来执行该笔支付。

决策的优化目标包含三个维度，需要在它们之间取得平衡：

$$
\max_{c_t} \quad \alpha \cdot g_{fee}(c_t, a_t) + \beta \cdot g_{balance}(c_t, a_t, B_t^c) + \gamma \cdot g_{time}(c_t, Q_t^c)
$$

三个维度的具体定义如下。**费率得分** $g_{fee}$ 衡量选择某通道的成本效益，费率越低得分越高，采用Sigmoid函数进行归一化：

$$
g_{fee}(c, a) = \frac{1}{1 + \exp((f_c(a) - \mu_f) / \sigma_f)}
$$

其中 $\mu_f = 2.0$、$\sigma_f = 1.0$ 为经验参数，将费率映射到 $(0, 1)$ 区间。**头寸得分** $g_{balance}$ 评估通道余额的充足程度，余额与支付金额的比值越大越安全：

$$
g_{balance}(c, a, B) = \begin{cases} \min(B / (10a), 1) & B \geq a \\ -1 & B < a \end{cases}
$$

当通道余额低于支付金额时，给予**负分惩罚**（-1），确保系统不会选择可能因头寸不足而导致支付失败的通道。**时效得分** $g_{time}$ 反映通道的拥堵程度，排队长度越短得分越高：

$$
g_{time}(c, Q) = \frac{1}{1 + 0.1 \cdot Q}
$$

权重参数 $(\alpha, \beta, \gamma) = (0.40, 0.35, 0.25)$ 的设定体现了业务优先级：费率成本是最重要的考量（40%），头寸安全次之（35%），客户体验时效性再次之（25%）。这些权重可根据银行风险偏好和经营策略进行动态调整——例如在月末头寸紧张时期，可适当提高 $\beta$ 的权重。

### 2.2 硬性约束过滤

在评分之前，系统首先执行**硬性约束过滤**，将不可用的通道从候选集中剔除：

$$
\mathcal{C}_{eligible}(p_t) = \{c \in \{HVPS, IBPS, CITY\} \mid \mathbb{1}_{available}(c, p_t) = 1\}
$$

可用性判断函数 $\mathbb{1}_{available}$ 综合了三个通道的业务规则：

$$
\mathbb{1}_{available}(c, p) = \begin{cases} 1 & c = HVPS \text{ (无硬性限制)} \\ \mathbb{1}_{[0, 100)}(a) & c = IBPS \text{ (金额<100万)} \\ \mathbb{1}_{city\_list}(b) & c = CITY \text{ (城商行白名单)} \end{cases}
$$

这一过滤机制确保任何违反业务规则的通道选择都不会进入评分环节，是保障系统合规性的第一道防线。

### 2.3 核心算法实现

以下是路由决策引擎的核心Python实现，包含通道费率计算、可用性检查和加权评分三个核心模块：

```python
from dataclasses import dataclass
import numpy as np

@dataclass
class Payment:
    """支付请求数据结构"""
    id: int
    amount: float          # 金额(万元)
    counterparty_bank: str # 对手行
    is_city_bank: bool     # 是否为城商行
    timestamp: int         # 时间戳(分钟)

@dataclass  
class ChannelState:
    """通道实时状态"""
    name: str
    balance: float         # 剩余头寸(万元)
    queue_length: int      # 当前排队长度
    avg_process_time: float # 平均处理时间(秒)
    is_available: bool     # 是否可用

def get_fee_hvps(package: int, amount: float) -> float:
    """大额系统单笔费率 (分段阶梯)"""
    tiers = {
        1: [(0, 20, 3.89), (20, 100, 4.59), (100, 1000, 4.69), (1000, float('inf'), 4.89)],
        2: [(0, 20, 3.69), (20, 100, 4.29), (100, 1000, 4.39), (1000, float('inf'), 4.59)],
        3: [(0, 20, 3.19), (20, 100, 3.59), (100, 1000, 3.79), (1000, float('inf'), 4.09)],
        4: [(0, 20, 2.29), (20, 100, 2.99), (100, 1000, 3.19), (1000, float('inf'), 3.49)],
        5: [(0, 20, 1.79), (20, 100, 2.09), (100, 1000, 2.29), (1000, float('inf'), 2.79)],
    }
    for lo, hi, fee in tiers[package]:
        if lo <= amount < hi:
            return fee
    return tiers[package][-1][2]

def calculate_score(payment, channel, state, 
                    w_fee=0.4, w_balance=0.35, w_time=0.25) -> float:
    """
    三维度加权评分函数
    Returns: 总分 (越高越好)
    """
    # 1. 费率得分 (低费率 -> 高分)
    fee = channel_fee(channel, payment.amount)
    fee_score = 1.0 / (1.0 + np.exp((fee - 2.0) / 1.0))
    
    # 2. 头寸充足度得分
    ratio = state.balance / max(payment.amount, 0.01)
    balance_score = min(ratio / 10.0, 1.0)
    if state.balance < payment.amount:
        balance_score = -1.0  # 头寸不足惩罚
    
    # 3. 时效得分 (短队列 -> 高分)
    time_score = 1.0 / (1.0 + 0.1 * state.queue_length)
    
    return w_fee * fee_score + w_balance * balance_score + w_time * time_score

def route_payment(payment, states, weights=(0.4, 0.35, 0.25)):
    """
    单笔支付路由决策主函数
    Returns: (selected_channel, score)
    """
    best_channel, best_score = None, -float('inf')
    
    for ch_name, state in states.items():
        # 硬性约束检查
        if not is_channel_eligible(ch_name, payment):
            continue
        if not state.is_available:
            continue
        # 评分
        score = calculate_score(payment, ch_name, state, *weights)
        if score > best_score:
            best_score = score
            best_channel = ch_name
    
    return best_channel, best_score
```

## 4. 头寸预测与资金配置优化

### 4.1 预测模型设计

通道头寸的动态管理是支付路由优化的"后勤战场"。银行每日需要在三个通道中预存足够资金以确保支付畅通，但资金过多会造成**机会成本浪费**（资金闲置在清算账户无法产生收益），资金过少则可能导致**支付失败**（头寸不足）。因此，准确预测未来一段时间内的头寸消耗量，是优化资金配置的前提。

我们构建了基于**梯度提升树（GBDT）**的时间序列预测模型，以分钟级粒度滚动预测各通道的未来头寸消耗。模型的输入特征包括：**时间特征**（日内分钟数、小时、是否早晚高峰）、**日历特征**（日期、是否月末、星期几）、**滞后特征**（前一时段的头寸消耗量）以及**外部特征**（节假日标记、特殊事件标记）。

| 特征类别 | 具体特征 | 说明 |
|---------|---------|------|
| 时间特征 | minute, hour | 日内时间位置 |
| 峰期标记 | is_morning_peak, is_afternoon_peak | 业务高峰时段指示 |
| 日历特征 | day_of_month, day_of_week | 日期和星期 |
| 月末效应 | is_month_end | 月末(25日+)标记 |
| 滞后特征 | consume_hvps_lag1, consume_ibps_lag1, consume_city_lag1 | 前一时段消耗量 |
| 业务因子 | time_factor | 双峰时间模板值 |

模型在90天历史数据上训练，三个通道的预测性能如下：大额系统(HVPS) 测试集 $R^2 = 0.872$、超网系统(IBPS) 测试集 $R^2 = 0.857$、城商行通道(CITY) 测试集 $R^2 = 0.736$。大额和超网通道的预测精度较高，主要因为它们的业务量较大、时间模式稳定；城商行通道的预测精度稍低，原因在于其业务量较小、随机波动相对更大。

![头寸预测分析](fig_position_forecast.png)

左上角的日内消耗模式图揭示了典型的**双峰特征**：上午10:00左右出现第一个高峰（HVPS峰值约32万元/10分钟），下午14:30左右出现第二个高峰。这一模式与银行对公业务的实际作息高度吻合——上午是企业集中发起支付指令的时段，下午则是跨行转账的第二个高峰。右上角的预测-实际对比图显示，GBDT模型能够较好地捕捉消耗的波动趋势，但在极端峰值处存在一定滞后。左下角的特征重要性分析表明，**time_factor**（双峰时间模板）是最重要的预测因子，其次是**day_of_month**（月末效应）和各通道的**滞后消耗量**。右下角的月末效应图清晰显示，从25日开始三通道的消耗量均有明显上升趋势，HVPS通道的月末增幅可达**25-30%**。

### 4.2 日终头寸规划

基于预测模型，日终头寸规划可形式化为一个**带机会成本约束的优化问题**。设次日预测的总头寸消耗为 $\hat{D}_c$（由预测模型输出），机会成本率为 $r$（通常为隔夜利率或内部资金转移价格），安全垫系数为 $\theta$（通常取1.2-1.5，对应20%-50%的缓冲），则最优初始头寸配置为：

$$
B_c^{init} = \theta \cdot \hat{D}_c + Q_c^{max} \cdot \bar{a}_c
$$

其中 $Q_c^{max}$ 为通道最大排队长度，$\bar{a}_c$ 为平均单笔金额。第二项 $Q_c^{max} \cdot \bar{a}_c$ 是应对突发高峰的**缓冲头寸**，确保在极端情况下队列中的支付也能全部完成。总资金成本的优化目标为：

$$
\min \sum_c \left[ r \cdot B_c^{init} + \lambda \cdot \mathbb{1}_{B_c^{init} < D_c^{actual}} \cdot P_{penalty} \right]
$$

第一项是资金的机会成本，第二项是头寸不足的惩罚成本（$P_{penalty}$ 可设为支付失败导致的客户投诉成本和紧急调拨成本），$\lambda$ 为风险偏好系数。通过调整 $\theta$ 和 $\lambda$ 的取值，银行可以在"资金效率"和"支付安全"之间找到适合自身风险偏好的平衡点。

## 5. 系统架构与部署方案

### 5.1 五层闭环架构

![系统架构图](fig_system_architecture.png)

系统采用**五层闭环架构**设计，各层职责明确、数据流向清晰。**数据输入层**负责接收实时支付请求、历史流水数据、外部日历数据和通道实时状态四路数据源。**旁路预测层**是系统的"大脑"，以分钟级频率运行GBDT预测模型、异常检测模块和费率优化计算，为决策层提供前瞻性的数据支撑。**智能决策层**是核心路由引擎，包含硬性约束过滤器和多目标加权评分器两个子模块。**通道执行层**对接三套支付清算系统的API，完成实际的支付报文发送和头寸扣减。**反馈优化层**负责监控路由效果、评估模型性能，并在必要时触发头寸预警和自动调拨。

整个架构的关键设计原则是**旁路化**——预测和优化计算作为旁路系统运行，不阻塞主支付流程的实时性。支付请求的端到端处理延迟目标控制在**50毫秒以内**，其中路由决策本身的计算耗时不超过**5毫秒**。

### 5.2 部署与演进建议

| 阶段 | 周期 | 目标 | 关键任务 |
|------|------|------|---------|
| **第一阶段** | 1-2个月 | 旁路试运行 | 部署预测模型，输出建议但不自动执行；积累A/B测试数据 |
| **第二阶段** | 3-4个月 | 灰度自动路由 | 对20%流量启用自动路由决策；监控成功率和客户投诉 |
| **第三阶段** | 5-6个月 | 全量上线 | 100%流量自动路由；建立日终头寸规划的自动化流程 |
| **第四阶段** | 6-12个月 | 智能升级 | 引入强化学习在线优化权重；构建通道堵塞的主动预测能力 |

在模型维护方面，建议建立**月度模型重训练**机制，利用最新的支付流水数据更新GBDT预测模型。同时设置**模型漂移监控**指标（如预测误差MAPE连续3天超过阈值20%），触发紧急模型更新。费率结构的变更（如央行调整清算费率、套餐档位变化）需要及时同步到费率计算模块，建议通过配置化方式管理费率表，避免硬编码。

## 6. 关键结论与业务价值

本方案为银行支付通道管理提供了一套完整的**数据智能解决方案**。核心创新点在于将运筹学的多目标优化框架与机器学习的预测能力相结合，实现了从"静态规则匹配"到"动态智能决策"的跨越。

| 优化维度 | 优化前(规则匹配) | 优化后(智能路由) | 预期改善 |
|---------|----------------|----------------|---------|
| 通道费率成本 | ~7,000元/天 | ~3,800元/天 | **节省45-50%** |
| 头寸利用率 | ~60% | ~85% | **提升25个百分点** |
| 支付成功率 | ~99.2% | ~99.8% | **提升0.6个百分点** |
| 客户投诉率 | ~0.15% | ~0.05% | **降低67%** |

从经济学角度看，以年交易日250天计算，通道费用年节省额约为 **80-100万元**，同时头寸效率的提升可释放约 **20-30%的备付资金** 用于高收益资产配置，按当前市场利率估算可额外创造 **50-80万元/年** 的资金收益。两项合计，年度直接经济效益可达 **130-180万元**，而系统的建设和运维成本（以云服务+人力估算）约为 **30-50万元/年**，投资回报率（ROI）超过 **200%**。

更重要的是，这一方案为银行支付基础设施的**数字化转型**奠定了方法论基础。路由优化引擎的框架具有良好的可扩展性——未来新增第四、第五条支付通道时，只需补充相应的约束规则和费率函数即可平滑接入；权重参数的在线学习能力可随着业务环境的变化自动调整策略；预测模型的特征工程框架也可迁移至其他资金预测场景（如备付金管理、流动性覆盖率预测等）。
