## 2.2 Unicode Encodings
名词解析
Unicode standard
	这个是Unicode 标准，它囊括几乎世界上所有的文字，符号，表情。Unicode 是**一本字典**，  给全世界的每个字符分配了一个唯一的编号。
code points
	code points 码点。是每个字符的唯一编号。而编号的存储形式，就是UTF-8 或者UTF-16 等
UTF-8 UTF-16 UTF-32

我们有了 Unicode standard 标准，我们能找到每个字符的唯一表示。
那为什么不把用 这个 唯一标识 也就是 code point 来直接编码字符呢？
如果直接把每个 code point 当成一个 token。词表大小 ≈ 150,000。实际常用字符 < 10,000。而大部分字符（如冷门汉字、符号、emoji）几乎不出现结果是一个**超大且稀疏的词表**，  
训练时会非常浪费显存、参数、梯度更新都不稳定。

Tokenizer 需要把文本拆成“词元（token）”。
- 如果以 code point 为单位：
    - 每个字符一个 token → 150k 维 one-hot 向量。
    - 超大、稀疏、很多根本不用。
- 如果以 UTF-8 字节为单位：
    - 所有可能的字节值只有 0~255 共 256 种。
    - 小而稳定，每个字符可以由 1~4 字节组成。
如果直接用code point。那么tokenizer 要记住每一个映射关系。
但是如果我用 UTF-8
比如说 
字符 `'中'` 的 UTF-8 编码是： `E4 B8 AD   （3 个字节）`
tokenizer 就可以把 `'中'` 当成三个小块拼成的序列：`[E4] [B8] [AD]`
就可以只用 256 个基础字符，拼出所有字符。

## 2.3 Subword Tokenization 
子词分词
在 byte-level tokenization (字节级分词)中，会将字符 拆成 UTF-8 字节序列，然后用二进制保存。
优点是绝对不会有 OOV的情况发生。

字节级别分词的结果会很长。模型在处理输入信息的时候，由于输出长度很长，模型会处理 long-term dependency。这样会让模型很难处理。好处是词表是固定的，而且很小

而 word level tokenization 词级分词 会把每个词语当成一个token。会有OOV的现象，词表会变得很大。

sub-word tokenization 词级别和 字节级别分词的折中选择。
这个方法依赖于 BPE算法来构建词表，而不是和 word level tokenization 的方法一样。
BPE 会从最基本的子词或者字节出发(从子词出发就叫做Subword-level BPE，BERT，T5用这个,从字节出发就是 byte-level BPE GPT用这个)。反复合并预料中最常出现的相邻token 对，然后给这些相邻token对分配索引。通过这种方式建立起词表

## 2.4 BPE tokenizer Training 
**vocabulary initalization** 词表初始化。在训练之前，我们先将词表进行初始化，然后再一个一个地合并token，最后得到结果。
我们最初的词表 一开始只有字节 0-255 每个字节都是一个token，后面我们通过BPE，将常见的 字节对形成新的token。
大概是
```python
vocab = {
	b'\x00':0, b'\x01':1,b'\x02':2...., b'\xff': 255
}
```

**pre-tokenization**  方法。
当你有了初始化的词表之后，你就可以开始计算 `how often bytes occur next to each other in your text.`  多少个 bytes对 出现在 文本中。然后把这些文本对合并，给他们一个索引。
但是这样做是很耗费计算资源的，因为在合并的时候，你不得不重复传入词汇表。
```python
# 以 'text' 为例
vocab = initialize_vocab() # 初始情况 256 个 bytes
for step in range(num_merges):
	pair_counts = count_adjacent_pairs(corpus,vocab)
	best_pair = find_most_frequent(pair_counts)
	vocab.add(best_pair) # 这里其实是替换
```
在这输入语料和词表，来统计所有相邻token 对出现的频率，然后找到出现次数最多的那个对。
并把它加到 词表里面。

所以这就是为什么每次都会重复地传入词汇表。
另外，直接对整个文本做BPE来生成词表，会生成许多几乎相同的token。这些token只差一个标点符号。比如 `dog!` 和 `dog.` 会被认为是两个不同的token，然后被记录在词表中。

为了避免这些，pre-tokenization 的方法就被发明出来了。可以把这个方法想象为是一个 粗粒度的tokenization。具体做法是 先把文本切分成较大的单位，如子词，然后统计每个单词出现的频率。最后在每个单词内部，再去统计相邻字符的出现次数，用于后续的 BPE合并。
以`text` 为例，可能他出现了10次，然后其中 `te`,`ex`, `xt` 个一次，选te为次数最多的。然后计算一下。那么 `te`就出现了10次。然后更新词表把 `te`  加进去(替换)
